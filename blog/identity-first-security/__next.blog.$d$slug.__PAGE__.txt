We spent three years rolling out Zero Trust at a defense technology company. 57 tasks later, here's what actually worked — and what broke in production.

## The Implementation Sequence

Zero Trust isn't a product you buy. It's a sequence of controls you enforce, in order, with each phase creating the foundation for the next.

### Phase 1: Emergency Access & Monitoring Foundation

Before touching anything else, you need:

- **Breakglass accounts** — Named emergency accounts that bypass all Conditional Access. Don't use these daily. Test them quarterly.
- **Alerting baseline** — What does normal look like? Capture login patterns, device health, app usage before you start blocking.

This is the unsexy phase that saves you when Phase 3 locks everyone out.

### Phase 2: Security Info Registration

Before enforcing MFA, users need to register authentication methods. Set this up first:
- **Auth methods policy** — Enable Microsoft Authenticator, FIDO2, Windows Hello for Business
- **Temporary Access Pass** — One-time codes for users who lose hardware keys
- **Report-only mode** — Test CA policies before enforcing

### Phase 3: Block Legacy Authentication

Legacy auth (SMTP auth, basic auth, POP/IMAP) is the attack surface. Block it:

```
Conditional Access > Assignments > Cloud apps > All apps
Conditions > Client apps > Select Exchange ActiveSync clients and Other clients
Controls > Grant > Block access
```

Block legacy auth too early and you'll break:
- Legacy application integrations
- Service accounts with password auth only
- Old MFA integrations that sit behind legacy protocols

**What broke first:** SMTP-authenticated printer scanning and IoT device integrations. Both used basic auth with no modern auth path.

### Phase 3: MFA Enforcement

Phishing-resistant MFA is the goal. FIDO2 hardware keys. But hardware keys have procurement and onboarding friction:

- Procurement: Buy 50 keys, distribute to Tier-1 roles (privileged users, admins)
- Onboarding: Users need to register in-browser with Chrome/Edge (Firefox support is spotty)
- Policy: Require phishing-resistant MFA for admin roles; offer standard MFA as fallback for general users

**What broke first:** Users who lost their hardware keys. Registration flow required a new key + backup key, and the process times out fast.

### Phase 4: Device Compliance (Intune Baselines)

Intune security baselines + Conditional Access is where device trust lives:

- **Windows baseline** — BitLocker, Windows Defender, firewall rules
- **Device compliance policy** — Require Microsoft Entra hybrid joined OR compliant device
- **MDE integration** — Device risk from Defender feeds CA decisions

**What broke first:** Microsoft Entra hybrid join fails when:
- Stale device objects in on-prem AD aren't cleaned up
- Line-of-business apps still query on-prem AD only (no cloud auth path)
- Users are co-managed (Intune + SCCM) and the policy boundary isn't clear

### Phase 5: Session Controls

Sign-in frequency and device restrictions for existing sessions:
- **Sign-in frequency** — Re-authenticate after X hours or when device state changes
- **Persistent browser** — Block to prevent session cookie replay
- **App-enforced restrictions** — Disable clipboard/file copy for unmanaged devices

### Phase 6: Risk-Based Conditional Access

Identity Protection risk signals drive dynamic policies:

- **High-risk sign-in** — Block or require MFA
- **Risky user** — Force password reset
- **Microsoft Defender for Identity** — Detect lateral movement, privilege escalation, and domain escalation attacks

**What broke first:** False positives. Risky sign-in policies flagged VPN users on corporate devices because the sign-in location didn't match.

### Phase 7: Network Segmentation (802.1X)

WiFi authentication via 802.1X (WPA2-Enterprise) is network-level Zero Trust:

- **EAP method** — EAP-TLS with certificate-based auth (vs. EAP-PEAP/MSCHAPv2)
- **Certificate lifecycle** — SCEP or PKCS certificate profile from Intune
- **Fallback** — WPA2-PSK for legacy devices that can't do 802.1X

**What broke first:** EAP negotiation failures. Root causes:
- Wrong EAP type configured on client vs. RADIUS
- Server certificate missing trusted root on client
- Server name validation mismatch (certificate SAN doesn't match RADIUS hostname)
- Expired or revoked client/server certificates

## What Actually Broke First

After three years, here's the friction ranking:

| Friction Point | Phase | Symptom |
|---|---|---|
| 802.1X EAP failures | Network | Users can't connect to WiFi; RADIUS shows method mismatch |
| Device baseline conflicts | Device | Co-managed devices have conflicting policies from SCCM and Intune |
| Legacy app breakage | Identity | LOB apps fail when legacy auth is blocked |
| False-positive risky sign-ins | Identity | VPN users flagged as risky based on location |
| Hardware key loss | MFA | Users lose keys, no backup, lockout |
| Breakglass account lockout | Foundation | Breakglass account passwords expire or are not tested |

## The Hard Lessons

1. **Don't start with blocking.** Start with alerting and monitoring. Baseline normal before you break normal.

2. **Phased enforcement, not big bang.** Each phase enables the next. Block legacy auth → enforce MFA → require device compliance → apply risk-based access.

3. **802.1X is the hardest network control.** Budget double the time for 802.1X pilot. EAP-TLS certificate validation is where it fails.

4. **Device trust requires clear ownership.** If you're co-managed (SCCM + Intune), define the boundary. We used Intune for cloud-native devices, SCCM for remaining on-prem managed.

5. **Governance is the bottleneck.** Zero Trust fails when cross-team ownership isn't clear. Identity owns CA policies. Endpoint owns device baselines. Network owns 802.1X. Security owns MDI.

## The Takeaway

Zero Trust isn't a project with an end date. It's an operational posture. The sequence matters: emergency access → security info registration → legacy block → MFA → device compliance → session controls → risk-based access → 802.1X.

Your users have personal Gmail accounts open on their work machines. They're checking personal email alongside Google Workspace. They're sharing work documents from personal Drive. IT has no visibility, no DLP, no audit trail.

This is the shadow IT problem. You can't just block personal Gmail at the network level without breaking something. But you can make it unusable for work purposes — and that's what this post covers.

## The Problem: Shadow IT in Google Workspace

Every organization has it: users who default to personal accounts for convenience. Personal Gmail on work devices. Personal Drive for sharing. Personal Calendar for everything.

The risks:
- **No DLP** — Company data leaves to personal accounts with no controls
- **No audit trail** — You can't investigate a data leak if you don't know it happened
- **No lifecycle** — When they leave, access to work data persists in their personal account
- **Compliance risk** — Often noncompliant with internal policy and regulated data-handling requirements

The goal: make personal Gmail unusable for work purposes without breaking legitimate personal use on personal devices.

## How CASB Fits In

CASB (Cloud Access Security Broker) sits between your users and cloud services, providing:
- **Visibility** — What cloud apps are being used?
- **Data protection** — DLP policies applied to cloud traffic
- **Threat protection** — Anomaly detection for cloud usage
- **Access control** — Block or restrict based on context

Google Workspace integrates with CASB via API, not inline proxy. Google explicitly recommends API/offline CASB patterns over inline proxies — inline CASBs exist but aren't the preferred path. For most orgs, Google-native controls (DLP, Reports API, App Access Management) cover much of what you need before considering third-party CASB.

### Shadow IT Discovery

Before you can block, you need to know what you're dealing with. The CASB (or Google Workspace audit logs) reveals:

- **OAuth token inventory** — Which third-party apps have user-authorized access to their Google data
- **Login activity** — Where are users authenticating from?
- **Drive sharing** — What's being shared externally?
- **App usage** — Which apps are accessing Google Workspace data?

In Google Workspace Admin console, go to Security > API controls > Access management. Review OAuth app entries — check ownership (Google-owned/internal/third-party), requested scopes, and user count. The CASB dashboard often surfaces this more visually.

Apps are categorized as **Trusted** (full access), **Limited** (specific Google data only), **Blocked** (no access), or the default for unconfigured apps. Mark unknown third-party apps as Blocked after reviewing scope and ownership.

## The Migration Strategy

### Phase 1: Discover and Categorize

List every external app accessing your users' Google data. Categorize:
- **Approved** — IT-sanctioned, security reviewed
- **Tolerated** — Not ideal but not blocked yet
- **Blocked** — Shadow IT, must go

The CASB dashboard usually provides this. If not, export audit logs and pivot in Excel.

### Phase 2: Offer a Better Alternative

Users go to personal Gmail because the sanctioned option isn't convenient. Fix that:

- **Google Workspace** — If you don't have it, get it. Every user gets company @company.com account.
- **Shared drives** — Centralize file storage so personal Drive isn't needed
- **SSO** — Single sign-on from Entra ID to Google Workspace removes the password burden

### Phase 3: Enforce with Policy

Now block. The enforcement stack, in order of impact:

**1. Block personal Google accounts** — This is what actually makes personal Gmail unusable on managed devices. Configure via proxy: `X-GoogApps-Allowed-Domains: yourdomain.com`, or in Google Admin console > Security > Managed browser > Restrict sign-in to managed accounts. This stops users from adding their personal @gmail.com account to Chrome or signing into personal Gmail on managed machines.

**2. App Access Control** — In Security > API controls > Access management, configure apps as Trusted / Limited / Specific Google data / Blocked. This controls OAuth grants, not browser access. Set default to Blocked for unconfigured apps if you want zero tolerance.

**3. Context-Aware Access** — Verify managed device before granting access to Workspace apps. Requires Endpoint Verification on desktop; mobile uses Google endpoint management (Intune, Jamf, etc.) for device signals.

### Phase 4: Data Migration

For users with existing personal Gmail data that needs to come over:

1. **Data import** — Admin imports personal Gmail to corporate account via Security > Data export and management > Data migration. Requires user authorization — they authorize their personal Gmail access. This isn't the "Google Workspace Migrate" tool (that's for Exchange migrations).
2. **Takeout** — User exports personal Drive, admin migrates to Shared Drive
3. **Calendar migration** — User exports from personal calendar, imports to corporate or shares calendar with corporate account

This takes time. Give users 30 days to migrate, then enforce.

## The Enforcement Checklist

| Control | Implementation | Effect |
|---|---|---|
| Consumer account block | X-GoogApps-Allowed-Domains header or managed browser sign-in restriction | Personal Gmail can't sign in on managed devices |
| App Access Control | Set apps to Trusted / Limited / Blocked; default to Blocked for unconfigured | No unapproved OAuth apps |
| Context-Aware Access | Require managed device (desktop: Endpoint Verification; mobile: endpoint management) | Unmanaged devices blocked |
| Drive external sharing | Allowlist or trust rules for external sharing | No uncontrolled file sharing |
| SSO | Entra ID to Google Workspace | One identity, one credential |

**CAA rollout tip:** Start in **monitor mode** first. Block only after verifying compliant devices sync Endpoint Verification — Google docs warn that sync delays can incorrectly block compliant desktops.

## What Actually Breaks First

In practice, this enforcement breaks these workflows:

1. **User's personal phone** — They can't check corporate email on their personal Android/iOS without management installed. Solution: offer MDM enrollment or accept the limitation.

2. **Third-party tools** — Users who authorized random SaaS apps lose access. Solution: pre-approve the most common ones (Zoom, Slack, etc.).

3. **Contractors** — External users without managed devices. Solution: dedicated contractor accounts with limited access, or accept unmanaged access with increased monitoring.

4. **Endpoint Verification sync delays** — Compliant desktops get blocked before they sync device status. Google's docs explicitly warn about this. Run in monitor mode first.

5. **OAuth restrictions break integrations fast** — Zoom, Slack, calendaring add-ons, scanners, backup tools — all need explicit app access configuration or users lose functionality.

6. **Delegated mailbox access** — Consumer account blocking with `X-GoogApps-Allowed-Domains` can break delegated mailbox access if the delegate's domain isn't in allowed headers.

## The Takeaway

You can't uninvent personal Gmail. Your users will always have personal accounts. What you can do is make it operationally useless for work:

- DLP policies block data exfiltration
- Context-Aware Access restricts access to managed devices
- SSO removes the password management burden
- Structured migration moves existing data to corporate

The goal isn't to punish users who use personal accounts.
