We spent three years rolling out Zero Trust at a defense technology company. 57 tasks later, here's what actually worked — and what broke in production.

## The Implementation Sequence

Zero Trust isn't a product you buy. It's a sequence of controls you enforce, in order, with each phase creating the foundation for the next.

### Phase 1: Emergency Access & Monitoring Foundation

Before touching anything else, you need:

- **Breakglass accounts** — Named emergency accounts that bypass all Conditional Access. Don't use these daily. Test them quarterly.
- **Alerting baseline** — What does normal look like? Capture login patterns, device health, app usage before you start blocking.

This is the unsexy phase that saves you when Phase 3 locks everyone out.

### Phase 2: Security Info Registration

Before enforcing MFA, users need to register authentication methods. Set this up first:
- **Auth methods policy** — Enable Microsoft Authenticator, FIDO2, Windows Hello for Business
- **Temporary Access Pass** — One-time codes for users who lose hardware keys
- **Report-only mode** — Test CA policies before enforcing

### Phase 3: Block Legacy Authentication

Legacy auth (SMTP auth, basic auth, POP/IMAP) is the attack surface. Block it:

```
Conditional Access > Assignments > Cloud apps > All apps
Conditions > Client apps > Select Exchange ActiveSync clients and Other clients
Controls > Grant > Block access
```

Block legacy auth too early and you'll break:
- Legacy application integrations
- Service accounts with password auth only
- Old MFA integrations that sit behind legacy protocols

**What broke first:** SMTP-authenticated printer scanning and IoT device integrations. Both used basic auth with no modern auth path.

### Phase 3: MFA Enforcement

Phishing-resistant MFA is the goal. FIDO2 hardware keys. But hardware keys have procurement and onboarding friction:

- Procurement: Buy 50 keys, distribute to Tier-1 roles (privileged users, admins)
- Onboarding: Users need to register in-browser with Chrome/Edge (Firefox support is spotty)
- Policy: Require phishing-resistant MFA for admin roles; offer standard MFA as fallback for general users

**What broke first:** Users who lost their hardware keys. Registration flow required a new key + backup key, and the process times out fast.

### Phase 4: Device Compliance (Intune Baselines)

Intune security baselines + Conditional Access is where device trust lives:

- **Windows baseline** — BitLocker, Windows Defender, firewall rules
- **Device compliance policy** — Require Microsoft Entra hybrid joined OR compliant device
- **MDE integration** — Device risk from Defender feeds CA decisions

**What broke first:** Microsoft Entra hybrid join fails when:
- Stale device objects in on-prem AD aren't cleaned up
- Line-of-business apps still query on-prem AD only (no cloud auth path)
- Users are co-managed (Intune + SCCM) and the policy boundary isn't clear

### Phase 5: Session Controls

Sign-in frequency and device restrictions for existing sessions:
- **Sign-in frequency** — Re-authenticate after X hours or when device state changes
- **Persistent browser** — Block to prevent session cookie replay
- **App-enforced restrictions** — Disable clipboard/file copy for unmanaged devices

### Phase 6: Risk-Based Conditional Access

Identity Protection risk signals drive dynamic policies:

- **High-risk sign-in** — Block or require MFA
- **Risky user** — Force password reset
- **Microsoft Defender for Identity** — Detect lateral movement, privilege escalation, and domain escalation attacks

**What broke first:** False positives. Risky sign-in policies flagged VPN users on corporate devices because the sign-in location didn't match.

### Phase 7: Network Segmentation (802.1X)

WiFi authentication via 802.1X (WPA2-Enterprise) is network-level Zero Trust:

- **EAP method** — EAP-TLS with certificate-based auth (vs. EAP-PEAP/MSCHAPv2)
- **Certificate lifecycle** — SCEP or PKCS certificate profile from Intune
- **Fallback** — WPA2-PSK for legacy devices that can't do 802.1X

**What broke first:** EAP negotiation failures. Root causes:
- Wrong EAP type configured on client vs. RADIUS
- Server certificate missing trusted root on client
- Server name validation mismatch (certificate SAN doesn't match RADIUS hostname)
- Expired or revoked client/server certificates

## What Actually Broke First

After three years, here's the friction ranking:

| Friction Point | Phase | Symptom |
|---|---|---|
| 802.1X EAP failures | Network | Users can't connect to WiFi; RADIUS shows method mismatch |
| Device baseline conflicts | Device | Co-managed devices have conflicting policies from SCCM and Intune |
| Legacy app breakage | Identity | LOB apps fail when legacy auth is blocked |
| False-positive risky sign-ins | Identity | VPN users flagged as risky based on location |
| Hardware key loss | MFA | Users lose keys, no backup, lockout |
| Breakglass account lockout | Foundation | Breakglass account passwords expire or are not tested |

## The Hard Lessons

1. **Don't start with blocking.** Start with alerting and monitoring. Baseline normal before you break normal.

2. **Phased enforcement, not big bang.** Each phase enables the next. Block legacy auth → enforce MFA → require device compliance → apply risk-based access.

3. **802.1X is the hardest network control.** Budget double the time for 802.1X pilot. EAP-TLS certificate validation is where it fails.

4. **Device trust requires clear ownership.** If you're co-managed (SCCM + Intune), define the boundary. We used Intune for cloud-native devices, SCCM for remaining on-prem managed.

5. **Governance is the bottleneck.** Zero Trust fails when cross-team ownership isn't clear. Identity owns CA policies. Endpoint owns device baselines. Network owns 802.1X. Security owns MDI.

## The Takeaway

Zero Trust isn't a project with an end date. It's an operational posture. The sequence matters: emergency access → security info registration → legacy block → MFA → device compliance → session controls → risk-based access → 802.1X.

Federating Google Workspace to Entra ID sounds straightforward — flip the switch in the admin console and suddenly your users sign in with Entra credentials. The reality is more nuanced. Here's what actually happens and how to get it right.

## Why Federate at All

The value isn't just SSO. It's consolidating your identity authority. When Entra is the IdP:

- **One identity** — Users exist in Entra, authenticate to Entra, access Google Workspace. No separate Google password to manage.
- **Conditional Access** — You can apply Entra CA policies before letting users into Google. MFA requirements, session limits, risk-based access — all enforced before Google sees the request.
- **Lifecycle** — When a user leaves Entra, they can't access Google. No orphaned accounts.

The alternative — local Google passwords with no Entra visibility — is the risk you're removing.

## The Architecture

Two systems, one federation contract:

```
User → Google Workspace (SP) → Entra ID (IdP) → SAML Response → User in Google
```

- **Entra ID** acts as the SAML Identity Provider
- **Google Workspace** acts as the Service Consumer (via Federated Directory)
- The glue is SAML 2.0 — standard, well-documented, but picky about configuration

The flow:

1. User visits `mail.google.com` or any Google property
2. Google detects federated domain, redirects to Entra with SAML AuthnRequest
3. Entra prompts for authentication (password + MFA if configured)
4. Entra returns SAML Assertion with user identity claims
5. Google validates assertion, creates session, user lands in Google Workspace

Two flows exist: SP-initiated (user starts at Google) and IdP-initiated (user starts at Entra, gets redirected to Google). Both work post-setup.

## The Setup Steps

### 1. Prepare Your Domains

Confirm both domains are verified in their respective consoles:
- In Entra, your primary domain must be verified
- In Google Admin, your domain must be listed

### 2. Create the Enterprise App in Entra

In Entra ID admin center:
- Add app > Google Workspace (from the gallery)
- Set up SSO > SAML
- **Important**: You'll populate this with values from Google in the next step — don't guess the endpoints.

### 3. Configure Google SSO Profile

In Google Admin > Security > Authentication > SSO with third-party IdP:
- Create a new SSO profile (or use the preconfigured Microsoft Entra profile if available)
- Copy these values from Google to Entra:
  - **Entity ID** (from Google)
  - **ACS URL** (from Google)
- Then populate Entra with:
  - **IdP Login URL**: Your Entra SSO URL
  - **IdP Logout URL**: Your Entra logout URL
  - **IdP Certificate**: Upload the Entra signing certificate

Assign the app to pilot users/groups.

### 4. Claims Mapping

Entra sends these claims to Google:

| Entra Claim | Google Field |
|---|---|
| email | Primary email |
| givenName | First name |
| surname | Last name |
| NameID | Must match Google user's primary email |

The critical mapping: **NameID must match the user's primary email address in Google**. If it doesn't, sign-in fails.

### 5. Provisioning (Optional But Recommended)

Federation handles **authentication** — it doesn't create Google accounts. Users must exist in Google Workspace to sign in.

Two paths:
- **Manual**: Pre-create users in Google before they authenticate
- **Automatic**: Enable Entra provisioning to Google (via Entra ID > Google app > Provisioning). This syncs users from Entra to Google — create, update, disable — on a regular schedule.

Most production deployments enable automatic provisioning to keep Google accounts in sync with Entra. Without it, you must manually create users in Google before they can sign in.

## What Breaks First

### NameID/Email Mismatch

The most common error post-setup: authentication fails at Google because the NameID claim doesn't match the user's primary email in Google.

Fix: Ensure the email claim from Entra matches exactly what's in the Google user's profile.

### Wrong ACS or Entity ID

If the SAML endpoints are mismatched between Entra and Google, the assertion gets rejected.

Fix: Verify Google provides the correct ACS URL and Entity ID, and those are exactly what's configured in Entra.

### Certificate Expiration

Entra's SAML signing certificate expires Yearly (check your app's certificate expiry). Put a calendar reminder 30 days before. When it expires, users can't sign in until you update the certificate in Google admin.

### User Not Provisioned

If a user exists in Entra but not in Google, and provisioning is disabled, authentication fails.

Fix: Enable automatic provisioning, or manually create users in Google before they attempt to sign in.

### Not Assigned to App

Sometimes forgotten: the user or their group isn't assigned to the Google app in Entra.

Fix: Assign the app to the pilot group in Entra > Enterprise App > Assignments.

## The Auth Flow in Plain English

```
Browser starts at mail.google.com 
    ↓
Google sees federated domain, redirects (SAML AuthnRequest) to Entra
    ↓
User enters credentials at Entra (MFA triggers here if enabled)
    ↓
Entra validates, issues SAML Assertion with email + name + ImmutableId
    ↓
Google receives assertion, matches ImmutableId to Google user, logs in
```

The key insight: the user never enters a Google password. Entra owns the credential entry, password complexity, MFA enforcement. Google just trusts Entra's assertion.

## What You Gain

| Before Federation | After Federation |
|---|---|---|
| Users manage Google passwords separately | Single Entra password |
| No Entra visibility into Google access | CA policies apply before Google |
| Orphaned Google accounts on leavers | Access cuts off when Entra disables |
| No unified session policy | Entra controls session lifetime |

## The Takeaway

The setup isn't hard — it's exacting. Get the SAML metadata right, map ImmutableId correctly, enable provisioning, and test both SP and IdP flows. Once federated, you've moved the authentication authority to Entra and consolidated your identity stack.

Don't skip the test phase.
